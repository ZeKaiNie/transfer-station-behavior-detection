# 方向丁：恶意 API 中转站投毒的客户端侧检测（研究设计 v0，2026-07）

> 状态：**设计草案，待用户过目**。尚未跑真实 API。本文是启动 pilot 前的"一页设计"，
> 用于锁定威胁模型、检测协议、基线、指标和相对已有工作的 delta。
> 学术诚信：本文所有指标位均为**占位**，只有真实跑出来的数字才会填入并标注来源。

## 1. 背景与动机（诚实版）

现在大量用户通过**第三方 API 中转站/代理**（one-api / new-api 系的 OpenAI 兼容代理）
访问前沿模型，或让 Codex / Claude Code 这类高权限编码 agent 走中转站。中转站在链路上
**终结 TLS、明文持有全部交互**，因此是天然的应用层中间人：它能悄悄改写 system prompt、
改写 agent 的 tool call、把依赖换成 typosquat 包、静默把请求降级到更弱/更便宜的模型、
或只在"不被审计时"才动手。

- 该威胁本身**已被 AEGIS（arXiv:2606.16358）系统提出**，但 AEGIS 的防御是
  **TEE 硬件飞地**——重、需 provider 配合、**普通用户用个便宜中转站根本部署不了**；
  AEGIS 还明说"现有客户端侧防御可被绕过"。
- MCP 供应链有 ShieldNet（2604.04426，网络层 guardrail + 检测器），但那是 MCP 工具层，
  不是"推理请求经过的中转站"这一层。
- **OSS 现状（2026-07 GitHub 实搜）**：搭中转站的开源项目成熟一大把（one-api/new-api 系
  738+）；但"验证 LLM API 响应是否被篡改"搜出来 `total=0`，"检测链路里被静默换模型"
  只有 2 个 ★0 玩具仓（且只做文本归因）；提示注入检测工具都在**提示/工具输出**层，
  没有一个针对**传输/中转站**层。→ 客户端侧、无 TEE 的中转站投毒检测在 OSS 基本空白。

**本方向的 delta（狭窄、诚实、可部署）：**
> 一个普通用户，**不依赖 TEE、不需要 provider 配合、纯客户端**，能在多大程度上检测出
> 自己用的中转站在投毒？**哪些篡改类别客户端能可靠检出、哪些本质上不可检出/易被绕过？**
> 这个"可检测边界"随模型、随篡改类别如何变化？

这是一篇**测量 / 边界刻画**论文，不是"又一个新防御机制"。卖点是 finding 和开源可复现
harness，不是宣称首次发现威胁。

## 2. 威胁模型

- **攻击者**：一个恶意或被攻陷的 OpenAI 兼容中转站，位于 agent 与上游官方模型之间，
  持有全部明文，可任意改写交付给 agent 的响应。
- **攻击者能力**（对齐 AEGIS 四类，我们用**自建合成代理**复现，不碰真实第三方服务）：
  - `TOOL_CALL_REWRITE`：改写 agent 要执行的 tool call——包名换 typosquat、网络目标换
    exfil 域名、或新增外发调用。
  - `INSTRUCTION_INJECTION`：在交付的助手文本里注入攻击者指令，诱导 agent 下一步作恶。
  - `SILENT_MODEL_SWAP`：静默把请求路由到更弱/不同模型（省成本或降质）。
  - `AUDIT_EVASION`：只在请求**不含**审计/测试标记时才篡改（naive 探测会漏）。
- **攻击者对检测策略的了解**：分两档评估——(a) **静态攻击者**（不知道客户端在检测）；
  (b) **自适应攻击者**（知道检测协议，主动规避）。自适应档是论文强度的关键。
- **攻击者不能做**：不能篡改客户端本地代码/本地静态校验逻辑；无法同时攻陷用户配置的
  **多个独立 provider 直连通道**（跨 provider 交叉核验的信任根）。这条假设要在 Limitation
  里写清楚——若用户所有通道都走同一个恶意中转站，跨 provider 核验失效。

## 3. 客户端侧检测协议（被测对象）

均为纯客户端、无 TEE、无需 provider 配合：

- **D1 跨 provider 一致性交叉核验**：对同一请求向 ≥2 个**独立** provider 取响应，比较
  安全相关字段（tool_call 目标 / 包名 / capability / 决策）是否一致；分歧则 flag。
  信任根 = "攻击者难以同时攻陷多个独立通道"。
- **D2 tool-call 不变量校验**（纯本地、零额外调用）：目标域名 allowlist、依赖名规范化 +
  typosquat 编辑距离、是否出现新的外发网络目标、能力越权。
- **D3 输出分布 / 模型指纹**：检测静默换模型——用响应统计 / 已知模型指纹判断"这真是我
  点名的那个模型吗"。（真实档需实测指纹可分性；OSS 几乎空白。）
- **D4 canary / 挑战探针**：对同一任务发**成对请求**（带 / 不带良性 canary 标记），比较
  安全相关动作差异，抓 `AUDIT_EVASION` 条件篡改。

## 4. 基线（对照，避免"跟自己比"）

- **B0 TLS-only**：什么都不做（现状，普通用户默认）。
- **B1 单 provider replay**：重发同一请求给同一（可能恶意）通道，看是否稳定——弱基线。
- **B2 纯 schema 校验**：只校验响应 JSON 结构合法，不查语义/目标。
- **B3 单一提示注入检测器**（借鉴 OSS，如 opencode-injection-guard 思路）：只扫文本注入。
- **相关工作对照（不复现硬件）**：AEGIS（TEE）、ShieldNet（MCP 网络层）作为 related work
  定性对比，**不假装复现 TEE 方案**。

## 5. 指标

- 每类篡改的**检出率**（TPR）；
- 良性流量**误报率 FPR**（跨 provider 核验在真实模型自然分歧下的 FPR 是关键风险点）；
- **攻击到达率**（危险动作存在且未被任一检测器 flag）——有 / 无检测器对比；
- **自适应可绕过性**：自适应攻击者下每个检测器还能挡多少；
- **成本 / 延迟**：D1/D4 需额外调用，量化每请求增量成本与延迟（普通用户可部署性）；
- **模型维度**：以上指标随 provider / 模型变化的曲线（吃"多模型 key"这个独家资源）。

## 6. 我们期待并诚实接受的 finding 形态

- **正向**：某些篡改（tool-call 改写、静默换模型、条件篡改）在客户端**可低误报检出**，
  说明"普通用户不靠 TEE 也能挡一部分"——补上 AEGIS 留的可部署性缝隙。
- **负向（同样是贡献）**：纯 `INSTRUCTION_INJECTION` 产出"貌似合理但被带偏"的动作，
  **无可信参考时客户端无法判定**；或跨 provider 核验在真实模型自然分歧下 FPR 过高不可用；
  或自适应中转站能规避 D1/D4。→ 这类边界结论恰好**量化印证 AEGIS "需要 TEE" 的必要性**，
  是干净的负向结果。
- mock 阶段已见的结构（透明规则，非写死）：D1/D2 抓 tool-call 改写与换模型、D4 抓条件
  篡改、纯文本注入抓不到、FPR=0。**这些只是机制层占位，真实数字必须云模型跑出来。**

## 7. 实验分期（省额度、先验后扩）

1. **P0 离线 mock**（已完成）：harness + 4 类篡改 + 4 检测器 + 单测，验证机制与流程。
2. **P1 小批 pilot**（待 2 家官方 key）：2–3 模型 × 少量任务，先看
   **跨 provider 核验在真实模型上的 FPR 是否可用 + 有没有干净的可检测/不可检测分界**。
   有 → 值得全力做；没有 → 及时止损或换角度。
3. **P2 扩展**（若 P1 有 finding）：扩模型覆盖、加自适应中转站、量化成本/延迟/绕过性。
4. **P3（可选、谨慎、匿名化）真实世界观测**：低价值可弃用 key 穿过真实中转站、只跑良性
   任务，用检测器看现实中转站是否篡改/换模型。有法律/伦理/可复现/key 暴露风险，单独评估。

## 8. 安全与合规红线

- 只做防御性研究；不攻击真实第三方服务；不做未授权中转站渗透测试。
- 中转站攻击只在**本地合成代理**复现；只用虚拟目标、fake secret、本地 sink。
- 不做通用越狱、不绕过真实模型安全对齐产危害内容。
- 不提交 secret / raw runs / 大数据集；真实结果与 mock 结果严格分离记录。
- 云实验逐次记录：provider、精确 model id、日期、prompt 版本、temperature/top_p/max_tokens/
  seed/timeout、原始输出、空/畸形响应计数、闭源更新局限；配开源权重锚点。
