## 0. 一句话方向

> **普通用户在不依赖 TEE、不需要模型厂商配合、纯客户端的条件下，能在多大程度上检测出自己

## 2. 为什么是这个方向（选题方法论 + 排除过程）

2026 年 Agent 安全已是成熟红海，**不存在"没人碰过的头牌威胁"**。单人小组唯一能赢的杠杆不是
"占住某个新载体"，而是：**执行严谨度 + 一个别人缺的资源 + 一个诚实且能被记住的发现**。

已逐一排查并**排除**的方向（都已被近期工作占满，附代表作）：
- **Agent Skill 运行时安全 / 新检测器**：已出 SoK 综述（2607.13987），
  攻击/防御/基准/生态全套齐（SkillScan 2602.06547、POISE 2606.07943、PhantomSkill 2606.19191、
  SkillMutator 2606.14154、Cloak&Detonate 2607.02357、SkillGuard 2606.03024、
  Skill-Inject"测量易感性"2602.20156 …）。→ 红海，且"测量易感性""隐蔽注入"都被占。
- **跨-App 上下文投毒攻击**：已被 Confused ChatGPT（2606.00485）提出。
- **provenance / 授权类防御**：AuthGraph（2605.26497）、ScopeGate（2606.28679）、
  SEAgent（2601.11893）做得更强、用标准 benchmark。
- **API 中转站 MITM 威胁本身 + TEE 防御**：AEGIS（2606.16358）已系统提出。
- **MCP 供应链**：ShieldNet / SC-Inject-Bench（2604.04426）。
- **纯"多模型跑对比"**：门槛太低，单独不成论文（跑对比不难，难的是跑出 finding）。

**留给我们的缝隙**（AEGIS 自己留的）：AEGIS 的防御是 **TEE 硬件飞地**——重、需 provider
配合、**普通用户用便宜中转站根本部署不了**，且 AEGIS 明说"现有客户端侧防御可被绕过"。
于是 AEGIS 没答、我们能答的问题就是第 0 节那句。

**OSS 现状（2026-07 GitHub 实搜佐证）**：搭中转站的开源项目成熟一大把（one-api/new-api 系
OpenAI 兼容代理，738+）；但"验证 API 响应是否被篡改"搜出来 **0 个**仓库，"检测链路里被静默
换模型"只有 2 个 ★0 玩具仓，提示注入检测工具都在**提示/工具输出**层、**没有针对传输/中转站
层**的。→ 客户端侧、无 TEE 的中转站投毒检测在开源界基本空白。

## 3. 威胁模型

- **攻击者**：恶意或被攻陷的 OpenAI 兼容中转站，夹在 agent 与上游官方模型之间，终结 TLS、
  持有全部明文，可任意改写交付给 agent 的响应。
- **四类篡改**（对齐 AEGIS，用**自建合成代理**复现，绝不碰真实第三方服务）：
  1. `TOOL_CALL_REWRITE`：改写 tool call——包名换 typosquat、网络目标换 exfil、或新增外发调用。
  2. `INSTRUCTION_INJECTION`：在交付文本里注入攻击者指令，诱导 agent 下一步作恶。
  3. `SILENT_MODEL_SWAP`：静默降级到更弱/更便宜模型。
  4. `AUDIT_EVASION`：只在请求**不含**审计/测试标记时才篡改（naive 探测会漏）。
- **攻击者对检测的了解**：分 (a) 静态攻击者、(b) **自适应攻击者**（知道检测协议主动规避）两档；
  自适应档是论文强度关键。
- **信任根与假设**：攻击者不能改客户端本地代码/本地校验；无法同时攻陷用户配置的**多个独立
  provider 直连通道**（这是跨 provider 核验的前提，必须在 Limitation 写清：若所有通道都走同
  一个恶意中转站，D1 失效）。

---

## 4. 被测的客户端侧检测协议（纯客户端、无 TEE、无需 provider 配合）

- **D1 跨 provider 一致性交叉核验**：同一请求发 ≥2 个**独立** provider，比较安全相关字段
  （tool_call 目标 / 包名 / capability / 决策）是否一致，分歧则 flag。
- **D2 tool-call 不变量校验**（纯本地、零额外调用）：目标域名 allowlist、依赖名规范化 +
  typosquat 编辑距离、新外发目标、能力越权。
- **D3 输出分布 / 模型指纹**：判断"这真是我点名的那个模型吗"，抓静默换模型。
- **D4 canary / 挑战探针**：对同一任务发成对请求（带 / 不带良性 canary），比较安全相关动作差异，
  抓条件性篡改。

---

## 5. 基线与对照

- **B0 TLS-only**（现状，什么都不做）；**B1 单 provider replay**；**B2 纯 schema 校验**；
  **B3 单一提示注入检测器**（借鉴 OSS 思路，只扫文本注入）。
- **相关工作定性对比**：AEGIS（TEE）、ShieldNet（MCP 网络层）——**不假装复现硬件方案**。

---

## 6. 指标

每类篡改**检出率(TPR)**；良性流量**误报率(FPR)**（D1 在真实模型自然分歧下的 FPR 是关键风险）；
**攻击到达率**（危险动作存在且未被 flag，有/无检测器对比）；**自适应可绕过性**；**成本/延迟**
（D1/D4 需额外调用，量化每请求增量）；以及以上随 **provider/模型**变化的曲线（吃多模型资源）。

---

## 7. 我们诚实接受的两种 finding

- **正向**：某些篡改（tool-call 改写 / 换模型 / 条件篡改）客户端**可低误报检出** → 普通用户
  不靠 TEE 也能挡一部分，补上 AEGIS 的可部署性缝隙。
- **负向（同样是贡献）**：纯指令注入产出"貌似合理但被带偏"的动作、无可信参考时客户端判不了；
  或 D1 在真实模型自然分歧下 FPR 过高不可用；或自适应中转站能规避 D1/D4。→ **量化印证
  AEGIS "需要 TEE" 的必要性**，干净的负向结果。

## 9. 实验分期（省额度、先验后扩）

1. **P0 离线 mock**：已完成（见 §8）。
2. **P1 小批 pilot**（需 ≥2 家独立官方 key）：先看 **D1 在真实模型上的 FPR 是否可用 + 有没有
   干净的可检测/不可检测分界**。有→全力做；没有→及时止损/换角度。
   - 只有 DeepSeek 时可先跑 D2/D3/D4（单通道）；D1 待第二家 key。
3. **P2 扩展**（P1 有 finding 后）：扩模型覆盖、加自适应中转站、量化成本/延迟/绕过性。
4. **P3（可选、谨慎、匿名化）真实世界观测**：低价值可弃用 key 穿过真实中转站、只跑良性任务，
   看现实中转站是否篡改/换模型。有法律/伦理/可复现/key 暴露风险，单独评估，非现在。

## 10. 云模型实验规范（可复现性硬要求）

闭源云 API **可以写进论文**（主流做法），但必须：钉带日期/版本的 model id（不用 latest）、
固定并公开全部解码参数（temperature/top_p/max_tokens/seed/timeout）+ 完整 prompt 原文、
**保存每次原始输出为 artifact**、记录调用日期与空/畸形响应计数、诚实写 Limitation。
**并配一个开源权重模型锚点**（可走 Together/Fireworks/DeepInfra/硅基流动等托管服务，无需显卡）
做"权重冻结、永久可复现"的对照。
- DeepSeek 当前有效模型：`deepseek-v4-flash`、`deepseek-v4-pro`（均为推理模型，
  **max_tokens 需 ≥2048**，否则思考过程吃光 token 返回空）。**不要再用旧别名 `deepseek-chat`。**

## 11. 安全与合规红线

复现，只用虚拟目标/fake secret/本地 sink；不做通用越狱、不绕过真实模型安全对齐产危害内容；

## 13. 诚实天花板与最大不确定性

SCI Q3 现实可达；CCF-C 有一搏（约 25–35%，靠模型覆盖广 + 一个反直觉发现 + 开源产物）；
SCI Q2 不现实。**成败真正取决于 P1 能否跑出一个值得记住的边界发现**——这是最大不确定性，
现在谁都无法保证。因此策略是：**先用很小额度跑 P1 验证有没有 finding，有再全力做。**
